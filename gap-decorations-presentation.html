Blink Architecture

The Scope

Gap decorations don't fit neatly into one Blink subsystem. They span all three layers of the rendering pipeline.

This means touching CSS parsing, computed style, layout algorithms, and paint code — each with its own owners and conventions.

STYLE CSS parsing • computed style • serialization
LAYOUT Gap geometry • intersection points • accumulator
PAINT Rendering • ink overflow • invalidation

April 2025

Accumulator Design

The naive approach recomputes intersections from scratch whenever geometry changes. Too expensive for complex layouts.

The accumulator pattern: walk items/tracks once, building intersection data incrementally. Applied to both flex (CL 6469189) and grid (CL 6460775).

Before: O(n²)
Recompute all intersections on every change
After: O(n)
Accumulate incrementally in a single pass

The Repeater Surprise

Lists & Interpolation

Properties accept lists: column-rule-width: 1px, 3px, 1px with auto repeaters.
We initially kept repeaters compact — never expand them.

Then interpolation arrived
CSSWG #12431 RESOLVED: "Define gap decoration animations using LCM methods"

Smart interpolation needed expanded lists to compute LCM alignment.
We had to rewrite the interpolation system — the V2 interpolation (Sep–Nov 2025).

Takeaways

What We Learned

Design

Spec and implementation must evolve together. Parallel development catches issues that sequential never would.

Implementation

Gap decorations touch style, layout, and paint. Be ready to learn unfamiliar areas of the codebase — and lean on reviewers who know them.

Working Group

In-person meetings (F2F, TPAC) produce breakthroughs that async discussion can't. The segments model came from a whiteboard, not a GitHub comment.
